Beispielszenario: Vertriebsprozessmodellierung zur Managementunterstützung

Visualisierung der Zusammenhänge zwischen Support-Bearbeitungsprozessen, beteiligten Komponenten und Auftraggebern der Dataport - Kunden.

Dataport interne Kunden (im Folgenden User genannt) nutzen eine Vielzahl an Tools und Anwendungen für ihre tägliche Arbeit. Diese unterschiedlichen Tools und Anwendungen stellen die User manchmal vor Herausforderungen im Gebrauch und somit benötigen die User schließlich Unterstützung von Supporteinheiten.

Hat ein User solch einen Supportfall, so kann dieser selbstständig über ein Dataport internes Selfservice Portal den Supportfall in ein Formular eingeben. In dem Formular werden bestimmte Fragen dem User zu seinem Supportfall gestellt. Z.B. Wie oft tritt der Supportfall auf? Wann hat es das letzte Mal funktioniert? Von wo und wie erfolgt die Einwahl? Außerdem muss eine Kurz- und Langbeschreibung des Supportfalles vom User erfolgen. Nachdem der User alle erforderlichen Informationen in das Formular eingetragen hat. Kann noch eine Anwesenheit bekanntgegeben werden. Anschließend wird das Formular eingereicht. Das Formular wird in ein Dataport internes Ticketsystem (ITSM) überführt, wodurch die First Supporteinheit per E-Mail informiert wird. Die First Supporteinheit prüft die Eingaben, sofern möglich wird dem User unverzüglich Abhilfe geschaffen und der User per E-Mail darüber in Kenntnis gesetzt. Falls die First Supporteinheit keine Möglichkeit der Abhilfe hat, wird der Second Supporteinheit das Ticket übergeben. Die Second Supporteinheit verschafft dem User Abhilfe und informiert per E-Mail den User.

Jederzeit hat der User die Möglichkeit über das interne Selfservice Portal den aktuellen Status seines Tickets einzusehen.

Beschreibung

Das Modell 1.0.0 bildet den Kernprozess eines Supportfalls ab: vom Erfassen der Störung über 1st/2nd-Level-Bearbeitung bis zur Benachrichtigung und dem Schließen des Incidents.

Ablauf (Hauptschritte)

  1. User-Pool: Supportformular mit Kurz-/Langbeschreibung senden.
  2. ITSM-Pool: Ticketanlage & Weiterleitung an First Support.
  3. First Support: Prüfung
    • lösbar: Lösung & Info an User, Ticket schließen
    • nicht lösbar: Übergabe an Second Support
  4. Second Support: Analyse, Behebung, Dokumentation.
  5. E-Mail-Benachrichtigungen an den User (Status/Abschluss).

Charakteristik

  • Fokus auf Standardfluss ohne Sonderpfade.
  • Kompakte, gut nachvollziehbare E2E-Transparenz.

Beschreibung

Das Modell 2.0.0 erweitert den Kernprozess um Authentifizierung, Wissensdatenbank-Abfragen (DBSM), Feedback-Schleifen und Timer-Events (z. B. Auto-Close).

Ablauf (Hauptschritte & Erweiterungen)

  1. User-Pool: Login/Authentifizierung, Formular inkl. Abbruchpfade.
  2. ITSM-Pool: Ticketanlage mit Validierungen (DB-Abgleich/DBSM).
  3. First Support: zusätzlich
    • Wissensdatenbank-Abfrage zur schnellen Lösungssuche.
    • Feedback-Schleife (Rückfragen/Bestätigung durch User).
  4. Second Support: detailliertere Eskalations-/Dokupfade (ITIL-nah).
  5. User-Rückmeldung zur Lösung (ja/nein).
  6. Timer-Events für SLA-/Prozess-Automatisierung.

Wesentliche Unterschiede zu 1.0.0

  • Vorangestellte Authentifizierung und valide Ticketanlage.
  • DBSM/Wissensdatenbank als Entscheidungshilfe im 1st-Level.
  • Feedback-Schleife vor dem Schließen.
  • Timer-Events (z. B. automatisches Schließen).
